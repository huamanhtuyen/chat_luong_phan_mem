Tài liệu bàn giao để chứng minh chất lượng phần mềm

Để kiểm soát và chứng minh chất lượng trong các dự án phần mềm (đặc biệt là phần mềm may đo cho doanh nghiệp), nhóm phát triển và kiểm thử thường phải bàn giao đầy đủ các tài liệu dưới đây:

Danh sách tài liệu chính

  1. 1
    Kế hoạch kiểm thử (Test Plan)

    Mô tả phạm vi kiểm thử, mục tiêu, phương pháp, công cụ, nguồn lực.

    Gắn với tiêu chuẩn (VD: IEEE 829, hoặc ISO/IEC/IEEE 29119).

    Xem ví dụ mẫu

    Ví dụ: Kế hoạch kiểm thử cho ứng dụng E-commerce

    Phần Nội dung
    Phạm vi kiểm thử Đăng ký, đăng nhập, thanh toán, quản lý giỏ hàng
    Mục tiêu Đảm bảo chức năng hoạt động đúng, không có lỗi nghiêm trọng
    Phương pháp Manual testing + Automation (Selenium)
    Công cụ JIRA, Postman, BrowserStack
    Nguồn lực 3 Tester, 1 Automation Engineer
  2. 2
    Chiến lược kiểm thử (Test Strategy)

    Tài liệu mức cao hơn, định nghĩa nguyên tắc chung: loại test, mức test (unit, integration, system, acceptance), môi trường test.

    Xem ví dụ mẫu

    Ví dụ: Chiến lược kiểm thử theo mức độ

    Nguyên tắc:

    • Unit Test: Tự động 80%
    • Integration: Manual + API test
    • System: End-to-end scenarios
    • Acceptance: UAT với khách hàng
  3. 3
    Đặc tả ca kiểm thử (Test Case Specification / Test Scenario)

    Danh sách chi tiết ca kiểm thử, điều kiện đầu vào, bước thực hiện, kết quả mong đợi.

    Có thể gắn trực tiếp với User Story (Agile) hoặc Requirement ID (Waterfall).

    Xem ví dụ mẫu

    Ví dụ: Ca kiểm thử đăng nhập

    ID Tên ca kiểm thử Điều kiện đầu vào Bước thực hiện Kết quả mong đợi
    TC_001 Đăng nhập thành công Email: user@test.com
    Password: 123456    		 1. Mở trang đăng nhập
    2. Nhập email
    3. Nhập password
    4. Click "Đăng nhập"    		 Chuyển đến trang dashboard
    TC_002 Đăng nhập sai password Email: user@test.com
    Password: wrongpass    		 1. Mở trang đăng nhập
    2. Nhập email
    3. Nhập password sai
    4. Click "Đăng nhập"    		 Hiển thị thông báo lỗi
  4. 4
    Ma trận truy vết yêu cầu (Requirement Traceability Matrix – RTM)

    Bảng mapping giữa Yêu cầu ↔ Test case ↔ Kết quả test.

    Giúp chứng minh tất cả yêu cầu đều được kiểm thử.

    Xem ví dụ mẫu

    Ví dụ: Ma trận truy vết yêu cầu

    Requirement ID Mô tả yêu cầu Test Case ID Kết quả Test Trạng thái
    REQ_001 Người dùng có thể đăng nhập TC_001, TC_002 Pass, Pass Covered
    REQ_002 Hiển thị danh sách sản phẩm TC_003, TC_004 Pass, Fail Partial
    REQ_003 Thanh toán online TC_005 Pass Covered

    Coverage: 85% (6/7 test cases pass)

  5. 5
    Kết quả chạy kiểm thử (Test Execution Report)

    Log, screenshot, video, hoặc xuất từ công cụ CI/CD (Jenkins, GitLab, Azure DevOps…).

    Ghi rõ test pass/fail, thời gian, người thực hiện.

    Xem ví dụ mẫu

    Ví dụ: Báo cáo thực thi kiểm thử Sprint 5

    Test Case ID Tên Test Case Kết quả Thời gian Người thực hiện Ghi chú
    TC_001 Đăng nhập thành công PASS 2024-01-15 10:30 Nguyễn Văn A -
    TC_002 Đăng nhập sai password PASS 2024-01-15 10:35 Nguyễn Văn A -
    TC_003 Thêm sản phẩm vào giỏ FAIL 2024-01-15 11:00 Trần Thị B Bug ID: BUG_001
    Tổng số test cases: 15
    Pass: 12
    Fail: 3
    Pass Rate: 80%
  6. 6
    Báo cáo lỗi (Defect Report / Bug Report)

    Mỗi lỗi có ID, mô tả, mức độ nghiêm trọng (Severity), độ ưu tiên (Priority), trạng thái (Open/In Progress/Closed).

    Bằng chứng lỗi: ảnh chụp màn hình, log hệ thống, video ghi lại thao tác.

    Xem ví dụ mẫu

    Ví dụ: Báo cáo lỗi hệ thống

    Bug ID Mô tả Severity Priority Status Assignee
    BUG_001 Nút "Thêm vào giỏ" không hoạt động trên mobile High High In Progress Dev Team
    BUG_002 Text bị cắt ở footer trên IE11 Medium Medium Closed UI Team
    BUG_003 API timeout khi load dữ liệu lớn High Critical Open Backend Team
    Bằng chứng đính kèm:
    • Screenshot lỗi trên mobile
    • Video ghi lại thao tác
    • Log hệ thống
    Thống kê:
    • Tổng bugs: 15
    • Open: 3
    • Closed: 12
  7. 7
    Test Evidence (Bằng chứng kiểm thử)

    Đây là tài liệu trọng yếu khi bàn giao, thường gồm:

    • Screenshot khi test thành công.
    • Log hệ thống xác nhận dữ liệu đúng.
    • File export kết quả chạy test tự động (JUnit, Cypress, Playwright…).
    • Video screen record (nếu yêu cầu).
    Xem ví dụ mẫu

    Ví dụ: Bằng chứng kiểm thử cho chức năng đăng nhập

  8. 8
    Báo cáo độ bao phủ (Coverage Report)

    Chứng minh mức C0 (câu lệnh), C1 (nhánh) đã đạt bao nhiêu %.

    Export từ công cụ như JaCoCo, Istanbul, Clover, SonarQube…

    Xem ví dụ mẫu

    Ví dụ: Báo cáo độ bao phủ code (JaCoCo)

    Biểu đồ độ bao phủ
    Chi tiết thống kê

    Lines 8,540 10,675 80%
    Branches 1,247 1,781 70%
    Methods 892 1,487 60%
    Classes 145 290 50%

    Generated by JaCoCo on 2024-01-15

  9. 9
    Báo cáo tổng hợp bàn giao (Test Summary Report)

    Tổng quan số lượng ca kiểm thử, pass/fail, defect mở/đóng, DRE, leakage, coverage…

    Đóng vai trò như "giấy chứng nhận chất lượng" khi bàn giao.

    Xem ví dụ mẫu

    Ví dụ: Báo cáo tổng hợp kiểm thử Sprint 5

    Kết quả thực thi test
    Pass: 120
    Fail: 30
    Thống kê chi tiết
    Tổng số test cases: 150
    Tests passed: 120 (80%)
    Tests failed: 30 (20%)
    Defects found: 25
    Defects closed: 22
    Code coverage: 75%
    DRE (Defect Removal Efficiency): 88%
    Kết luận bàn giao

    Hệ thống đã đạt tiêu chuẩn chất lượng với tỷ lệ pass 80%. Các defect còn lại đều ở mức độ thấp và không ảnh hưởng đến chức năng chính. Sẵn sàng bàn giao cho production.

    Checklist kiểm thử hồi quy & smoke test (nếu có)

Lưu ý

Tùy từng công ty và thị trường (Nhật, Mỹ, Ấn Độ…), danh sách tài liệu có thể thay đổi. Tuy nhiên, các tài liệu trên gần như là chuẩn chung để chứng minh chất lượng phần mềm và đáp ứng yêu cầu kiểm toán hoặc ISO/CMMI.
